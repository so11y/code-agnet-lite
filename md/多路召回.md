# 示例：用户说“登录有 bug”时，Agent 的正确流程

## 1. 用户输入

```txt
用户：登录有 bug
```

这句话信息很少，不能直接让模型猜原因，更不能直接改代码。

错误流程：

```txt
用户说“登录有 bug”
  ↓
模型猜：可能是 token 问题
  ↓
直接修改代码
```


---

## 2. 总流程标注

```txt
用户输入
  ↓
Task Analysis 任务分析                 [AI + 规则]
  ↓
Query Rewrite 查询扩展                 [AI + 规则 + 项目索引]
  ↓
Multi-Retrieval 多路召回               [工具 + 规则]
  ↓
Clean 数据清理                         [规则 + 系统]
  ↓
Deduplicate 去重                       [系统 / 规则]
  ↓
Rerank 重排                            [规则 + 可选 AI]
  ↓
Context Packing 上下文打包             [系统 + 可选 AI 摘要]
  ↓
Router 选择模式                        [AI + 规则]
  ↓
Planner 生成 intent                    [AI]
  ↓
Guard 执行前检查                       [规则 + 系统]
  ↓
Executor 调用工具                      [工具 + 系统]
  ↓
Observer 整理工具结果                  [系统 + 可选 AI 摘要]
  ↓
Update State 更新状态                  [系统]
  ↓
Reviewer 判断是否完成                  [AI + 规则]
  ↓
Build Next Context 构造下一轮上下文     [系统 + 可选 AI 摘要]
  ↓
继续下一轮 / 换策略 / 停止 / final
```

---

## 3. Task Analysis：任务分析

标注：

```txt
[AI + 规则]
```

作用：

```txt
判断用户这句话是什么类型的任务。
```

用户说：

```txt
登录有 bug
```

分析结果可以是：

```json
{
  "taskType": "debug",
  "domain": "login",
  "needTools": true,
  "needCodeContext": true,
  "missingInfo": [
    "具体报错",
    "复现步骤",
    "失败页面",
    "是否 401",
    "是否跳回登录页"
  ],
  "likelyAreas": [
    "登录页面",
    "登录 API",
    "token 存储",
    "用户状态",
    "请求拦截器",
    "路由守卫"
  ]
}
```

这一阶段不是查文件，也不是改代码，只是判断任务类型。

---

## 4. Query Rewrite：查询扩展

标注：

```txt
[AI + 规则 + 项目索引]
```

作用：

```txt
把用户短句扩展成多个可搜索 query。
```

用户原句：

```txt
登录有 bug
```

可以扩展成：

```txt
login
auth
token
signin
signIn
user store
router guard
permission
request interceptor
Authorization
refreshToken
401
403
登录
认证
鉴权
权限
路由守卫
登录接口
```

这些 query 的来源：

```txt
用户原话       [规则]
领域词典       [规则]
项目文件名     [工具 / 系统]
项目 symbol    [工具 / 系统]
报错信息       [工具 / 用户]
AI 语义扩展    [AI]
```

注意：

```txt
AI 可以生成候选 query，但最终要由系统过滤、去重、限制数量。
```

---

## 5. Multi-Retrieval：多路召回

标注：

```txt
[工具 + 规则]
```

作用：

```txt
用多种方式找相关信息，不只靠一种搜索。
```

常见召回方式：

```txt
文件名搜索       [工具]
代码关键词搜索   [工具]
向量搜索         [工具]
symbol 搜索      [工具]
依赖搜索         [工具]
测试文件搜索     [工具]
git diff 搜索    [工具]
报错日志搜索     [工具]
```

例如可能召回：

```txt
src/pages/login.vue
src/api/auth.ts
src/store/user.ts
src/utils/request.ts
src/router/guard.ts
tests/auth.test.ts
```

这一步主要是工具和规则，不是 AI 自己猜。

---

## 6. Clean：数据清理

标注：

```txt
[规则 + 系统]
```

作用：

```txt
过滤无效内容，保留有价值的信息。
```

过滤：

```txt
node_modules/
dist/
build/
coverage/
.git/
*.lock
*.min.js
*.map
```

保留：

```txt
文件路径
命中原因
关键函数
相关代码片段
行号
import/export 关系
```

清理后的结果示例：

```json
{
  "path": "src/pages/login.vue",
  "startLine": 20,
  "endLine": 85,
  "symbols": ["handleLogin", "loginForm"],
  "reason": "登录提交入口，调用 auth.login 和 user store"
}
```

---

## 7. Deduplicate：去重

标注：

```txt
[系统 / 规则]
```

作用：

```txt
同一个文件被多路召回时合并。
```

例如：

```txt
auth.ts 被 filename 命中
auth.ts 被 keyword 命中
auth.ts 被 dependency 命中
```

合并成：

```json
{
  "path": "src/api/auth.ts",
  "sources": ["filename", "keyword", "dependency"],
  "score": 0.93
}
```

规则：

```txt
同一个 path 合并
source 越多，相关性越高
score 重新计算
reason 合并
```

---

## 8. Rerank：重排

标注：

```txt
[规则 + 可选 AI]
```

作用：

```txt
判断哪些内容最应该放进 context。
```

对于“登录有 bug”，常见优先级：

```txt
1. 登录入口页面
2. 登录 API
3. 用户状态 / token 存储
4. 请求拦截器
5. 路由守卫
6. 相关测试
7. 最近修改文件
```

重排结果示例：

```json
[
  {
    "rank": 1,
    "path": "src/pages/login.vue",
    "reason": "登录入口页面，最直接相关"
  },
  {
    "rank": 2,
    "path": "src/api/auth.ts",
    "reason": "登录接口定义"
  },
  {
    "rank": 3,
    "path": "src/store/user.ts",
    "reason": "保存 token 和 userInfo"
  },
  {
    "rank": 4,
    "path": "src/utils/request.ts",
    "reason": "请求拦截器，可能影响 token / 401"
  },
  {
    "rank": 5,
    "path": "src/router/guard.ts",
    "reason": "登录态判断和跳转"
  }
]
```

---

## 9. Context Packing：打包上下文

标注：

```txt
[系统 + 可选 AI 摘要]
```

作用：

```txt
把召回结果整理成模型能看的上下文。
```

示例 context：

```txt
用户任务：
登录有 bug

任务分析：
这是一个 debug 任务。
当前缺少具体报错和复现步骤。
高相关方向包括：登录页面、auth API、token 存储、请求拦截器、路由守卫。

已召回的高相关文件：
1. src/pages/login.vue
   原因：登录入口页面
2. src/api/auth.ts
   原因：登录 API 定义
3. src/store/user.ts
   原因：可能保存 token 和 userInfo
4. src/utils/request.ts
   原因：请求拦截器，可能处理 Authorization / 401
5. src/router/guard.ts
   原因：登录态判断和路由跳转

当前缺失信息：
- 用户没有提供具体报错
- 不知道是登录失败、跳转失败，还是 401

建议下一步：
- 先读取 src/pages/login.vue
- 再读取 src/api/auth.ts 和 src/store/user.ts
- 暂时不要直接 edit
```

---

## 10. Router：选择模式

标注：

```txt
[AI + 规则]
```

作用：

```txt
判断接下来走 Direct / ReAct / ToT / AoT。
```

对于：

```txt
登录有 bug
```

通常选择：

```json
{
  "mode": "react",
  "reason": "这是需要读取文件、分析代码、可能运行测试的 debug 任务"
}
```

推荐规则：

```txt
简单问答       -> Direct
普通代码任务   -> ReAct
复杂方案比较   -> ToT
固定流程任务   -> AoT
```

---

## 11. Planner：生成下一步 Intent

标注：

```txt
[AI]
```

作用：

```txt
根据当前 context，决定下一步做什么。
```

第一轮可能输出：

```json
{
  "intent": "read",
  "tool": "read_file",
  "args": {
    "path": "src/pages/login.vue"
  },
  "reason": "先读取登录入口页面，确认登录流程和调用链",
  "confidence": 0.9
}
```

这里不要直接 edit。

因为当前证据不足，应该先 read。

---

## 12. Guard：执行前检查

标注：

```txt
[规则 + 系统]
```

作用：

```txt
防止模型乱调用工具。
```

检查内容：

```txt
是否危险命令
是否重复调用
是否 confidence 太低
是否直接 edit 但证据不足
是否连续 inspect
是否超出权限
```

示例：

```json
{
  "allow": true,
  "reason": "读取登录入口文件是安全且合理的下一步"
}
```

这一步建议用硬规则，不要依赖 AI。

---

## 13. Executor：调用工具

标注：

```txt
[工具 + 系统]
```

作用：

```txt
真正执行工具。
```

例如：

```ts
read_file("src/pages/login.vue");
```

Executor 只负责执行，不负责推理。

---

## 14. Observer：整理工具结果

标注：

```txt
[系统 + 可选 AI 摘要]
```

作用：

```txt
把工具原始结果整理成 observation。
```

例如：

```json
{
  "tool": "read_file",
  "path": "src/pages/login.vue",
  "success": true,
  "summary": "文件包含 handleLogin，调用 auth.login 后把 token 写入 userStore，然后跳转首页",
  "importantSymbols": ["handleLogin", "loginForm", "userStore.setToken"],
  "newFindings": [
    "登录入口调用 src/api/auth.ts 的 login",
    "登录成功后依赖 userStore.setToken",
    "登录后 router.push('/')"
  ],
  "suggestedNext": ["读取 src/api/auth.ts", "读取 src/store/user.ts"]
}
```

---

## 15. Update State：更新状态

标注：

```txt
[系统]
```

作用：

```txt
记录本轮发生了什么。
```

例如：

```json
{
  "round": 1,
  "readFiles": ["src/pages/login.vue"],
  "changedFiles": [],
  "noProgressRounds": 0,
  "history": [
    {
      "round": 1,
      "intent": "read",
      "tool": "read_file",
      "summary": "读取登录入口，发现调用 auth.login 和 userStore.setToken",
      "progress": true
    }
  ]
}
```

---

## 16. Reviewer：判断是否完成

标注：

```txt
[AI + 规则]
```

作用：

```txt
判断任务是否完成，是否继续，是否换策略。
```

规则判断：

```txt
测试通过了吗？
有没有改文件？
有没有连续失败？
有没有连续 inspect？
有没有达到最大轮数？
```

AI 判断：

```txt
当前信息够不够？
下一步是否应该读 auth.ts？
是否已经能形成可靠结论？
```

示例：

```json
{
  "done": false,
  "needContinue": true,
  "nextSuggestion": "继续读取 src/api/auth.ts 和 src/store/user.ts",
  "reason": "当前只知道登录入口，还不知道 API 返回结构和 token 保存逻辑"
}
```

---

## 17. Build Next Context：构造下一轮上下文

标注：

```txt
[系统 + 可选 AI 摘要]
```

作用：

```txt
把 state、history、observation 压缩成下一轮模型可读 context。
```

示例：

```txt
用户任务：
登录有 bug

已读文件：
- src/pages/login.vue

发现：
- 登录入口是 handleLogin
- handleLogin 调用 auth.login
- 登录成功后调用 userStore.setToken
- 登录后 router.push('/')

当前假设：
- 可能是 auth.login 返回结构不符合预期
- 可能是 userStore.setToken 没正确保存 token

下一步建议：
- 读取 src/api/auth.ts
- 读取 src/store/user.ts

约束：
- 还没有足够证据，不要 edit
```

然后进入下一轮。

## 19. 三句话总结

```txt
AI 负责判断：
任务类型、查询扩展、模式选择、下一步 intent、是否继续。
```

```txt
工具负责获取事实：
搜索文件、读取代码、查 git、跑测试、拿日志。
```

```txt
系统负责控制流程：
清理、去重、重排、打包 context、记录 state、防重复、防危险。
```

最终原则：

```txt
不要让模型直接猜 bug。
先召回事实，再打包 context，最后进入 ReAct Loop。
```
