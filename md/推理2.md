# 12. 更智能的思考：Intent、Context 与防止无效推理

前面讲的是 CoT、ReAct、ToT、AoT 这些推理 / 执行模式。

但是在真正写 Agent 的时候，还有一个很重要的问题：

```txt
模型会不会一直原地打转？
```

比如模型一直说：

```txt
我再检查一下
我继续分析一下
我再看看问题在哪里
```

但实际上：

```txt
没有读新文件
没有查新信息
没有修改代码
没有运行测试
没有产生新结论
```

这就是无效推理。

所以 Agent 需要一套更智能的状态判断机制。

核心是：

```txt
不要只看模型说了什么
要看这一轮有没有产生新的状态变化
```

---

## 12.1 给每一轮加 Intent

Intent 表示：

```txt
模型这一轮到底想干什么？
```

它不是用户意图，而是 Agent 每一轮的动作类型。

可以定义成：

```ts
type Intent =
  | "search"   // 搜索代码
  | "read"     // 读取文件
  | "edit"     // 修改文件
  | "test"     // 运行测试
  | "inspect"  // 只是检查 / 分析 / 再看看
  | "final"    // 最终回答
  | "ask_user" // 需要问用户
```

每一轮模型不能只输出自然语言，而是输出结构化结果：

```json
{
  "intent": "read",
  "tool": "read_file",
  "args": {
    "path": "src/auth.ts"
  },
  "reason": "需要查看登录 token 校验逻辑"
}
```

这样系统就能知道：

```txt
这一轮是 read
调用了 read_file
读取的是 src/auth.ts
```

而不是只能看到一句模糊的：

```txt
我再检查一下。
```

---

## 12.2 inspect 是什么

`inspect` 表示：

```txt
这一轮没有调用工具，只是在分析 / 检查 / 思考。
```

比如这些都可以归为 inspect：

```txt
我再检查一下
我继续分析一下
我需要进一步确认
我再看看问题在哪里
```

如果偶尔 inspect 是正常的。

但是如果连续多轮都是 inspect：

```txt
round 3: inspect
round 4: inspect
round 5: inspect
```

说明 Agent 很可能在空转。

这时候系统应该介入：

```txt
强制换策略
强制调用工具
要求输出 final
或者停止循环
```

---

## 12.3 Intent 会进入 Context

Intent 本身不是 context。

更准确的关系是：

```txt
intent -> 写入 state/history -> 压缩后进入 context -> 喂回模型
```

也就是：

```txt
模型输出 intent
  ↓
系统执行工具 / 或不执行工具
  ↓
生成 observation
  ↓
写入 state.history
  ↓
buildContext(state)
  ↓
下一轮模型根据 context 再决策
```

例如一轮记录：

```ts
{
  round: 3,
  intent: "inspect",
  tool: null,
  args: null,
  observation: "没有产生新信息",
  progress: false
}
```

下一轮给模型的 context 可以是：

```txt
当前任务：修复登录失败

最近动作：
1. search：搜索 login/auth/token，找到 auth.ts
2. read：读取 auth.ts
3. inspect：只分析了一轮，没有新信息

状态：
- 连续无进展轮数：1
- 已读文件：src/auth.ts
- 已修改文件：无
- 测试状态：未运行

约束：
- 如果继续，请选择 search / read / edit / test / final
- 不要继续空转式 inspect
```

这样模型下一轮就知道：

```txt
刚才已经空转过一次了
不能继续说“再检查一下”
需要产生新动作或给最终结论
```

---

## 12.4 State 和 Context 的区别

Agent 内部要区分两个东西：

```txt
State   = 系统完整保存的运行状态
Context = 给模型看的压缩状态
```

State 可以很完整：

```ts
type AgentState = {
  task: string;
  round: number;
  history: HistoryItem[];

  readFiles: string[];
  changedFiles: string[];
  executedCommands: string[];

  failedCount: number;
  noProgressRounds: number;
  repeatedActionCount: number;

  currentHypothesis?: string;
  confidence?: number;
};
```

Context 不应该太长，只保留关键摘要：

```ts
function buildContext(state: AgentState) {
  return `
任务：${state.task}

当前轮次：${state.round}

已读文件：
${state.readFiles.join("\n")}

已修改文件：
${state.changedFiles.join("\n")}

连续失败次数：${state.failedCount}
连续无进展轮数：${state.noProgressRounds}

最近动作：
${state.history.slice(-5).map(item => {
  return `- round ${item.round}: ${item.intent}, ${item.summary}`;
}).join("\n")}
`;
}
```

所以：

```txt
State 是完整数据库
Context 是喂给模型看的摘要
```

不要把所有历史、所有文件、所有日志都塞进 context。

## 12.5 如何判断是否有进展

不能只看模型有没有回答。

要看这一轮有没有带来新信息。

可以定义：

```ts
type ProgressSignal = {
  hasNewFile: boolean;
  hasNewError: boolean;
  hasCodeChange: boolean;
  hasNewObservation: boolean;
  testStatusChanged: boolean;
  confidenceChanged: boolean;
};
```

判断是否有进展：

```ts
function hasProgress(signal: ProgressSignal) {
  return (
    signal.hasNewFile ||
    signal.hasNewError ||
    signal.hasCodeChange ||
    signal.hasNewObservation ||
    signal.testStatusChanged ||
    signal.confidenceChanged
  );
}
```

如果没有进展：

```ts
state.noProgressRounds++;
```

如果有进展：

```ts
state.noProgressRounds = 0;
```

当连续无进展达到阈值：

```ts
if (state.noProgressRounds >= 2) {
  // 强制换策略，或者停止
}
```

## 12.6 如何判断一直在 inspect

每轮历史记录：

```ts
type HistoryItem = {
  round: number;
  intent: Intent;
  tool?: string;
  args?: Record<string, unknown>;
  summary: string;
  progress: boolean;
};
```

判断最近 3 轮是不是都在 inspect：

```ts
function isStuckInspecting(history: HistoryItem[]) {
  const recent = history.slice(-3);

  return (
    recent.length === 3 &&
    recent.every(item => item.intent === "inspect")
  );
}
```

使用：

```ts
if (isStuckInspecting(state.history)) {
  return {
    action: "stop",
    reason: "连续 3 轮都在 inspect，没有产生有效动作"
  };
}
```

或者更温和一点：

```ts
if (isStuckInspecting(state.history)) {
  return {
    action: "force_change_strategy",
    reason: "连续 inspect，需要改为 search/read/test/final 之一"
  };
}
```

## 12.7 如何判断一直重复调用工具

模型有时候不是一直 inspect，而是一直重复调用工具。

比如：

```txt
round 1: read_file("src/auth.ts")
round 2: read_file("src/auth.ts")
round 3: read_file("src/auth.ts")
```

这也是空转。

可以给每个 tool call 生成签名：

```ts
function actionSignature(item: HistoryItem) {
  return JSON.stringify({
    intent: item.intent,
    tool: item.tool,
    args: item.args ?? {}
  });
}
```

判断是否重复：

```ts
function isRepeatedAction(history: HistoryItem[], current: HistoryItem) {
  const currentSig = actionSignature(current);

  const count = history
    .slice(-5)
    .filter(item => actionSignature(item) === currentSig)
    .length;

  return count >= 2;
}
```

如果重复：

```ts
if (isRepeatedAction(state.history, current)) {
  return {
    action: "reject",
    reason: "最近已经执行过相同工具调用，请换策略"
  };
}
```

## 12.8 可信度 Confidence

除了 intent，还可以让模型输出一个可信度。

例如：

```ts
type AgentDecision = {
  intent: Intent;
  tool?: string;
  args?: Record<string, unknown>;
  reason: string;
  confidence: number; // 0 - 1
};
```

含义：

```txt
confidence = 模型对当前判断的把握程度
```

例如：

```json
{
  "intent": "edit",
  "tool": "edit_file",
  "args": {
    "path": "src/auth.ts"
  },
  "reason": "auth.ts 中 token 判断逻辑与测试失败信息一致",
  "confidence": 0.82
}
```

如果 confidence 很低：

```txt
不要直接 edit
优先 search / read / test 获取更多证据
```

例如：

```ts
if (decision.intent === "edit" && decision.confidence < 0.7) {
  return {
    action: "reject",
    reason: "修改代码前可信度不足，需要先读取更多上下文或运行测试"
  };
}
```

这样可以防止模型乱改代码。

## 12.9 什么时候换思考方式

Agent 可以根据 context 动态切换策略。

例如：

```txt
默认用 ReAct
如果连续无进展，切换 ToT
如果流程很固定，切换 AoT
如果信息足够，直接 final
```

可以定义：

```ts
type ThinkingMode = "react" | "tot" | "aot" | "direct";
```

切换规则：

```ts
function chooseThinkingMode(state: AgentState): ThinkingMode {
  if (state.noProgressRounds >= 2) {
    return "tot";
  }

  if (state.failedCount >= 2) {
    return "tot";
  }

  if (state.task.includes("按流程") || state.task.includes("每次都")) {
    return "aot";
  }

  return "react";
}
```

含义：

```txt
正常执行：ReAct
卡住了：ToT，生成多个新假设
固定流程：AoT
足够简单：Direct
```

## 12.10 卡住时为什么切 ToT

当 Agent 一直卡住时，说明当前单一路径不行。

比如：

```txt
一直读 auth.ts
一直检查 token
一直运行同一个失败测试
```

这时应该让模型切换为 ToT：

```txt
不要继续沿着同一条路
而是生成多个候选原因
```

例如：

```txt
当前卡住：登录测试仍然失败

请生成 3 个新的候选原因：
1. token 生成错误
2. token 存储错误
3. 请求拦截器没有携带 token

然后分别说明：
- 需要验证什么
- 应该读取哪个文件
- 应该运行哪个命令
- 哪个优先级最高
```

这就是：

```txt
ReAct 卡住 -> ToT 重新生成多条路径 -> 选一条继续 ReAct
```

## 12.11 防止无效推理的规则

可以总结成几条硬规则：

```txt
1. 连续 2 轮没有新信息：要求换策略
2. 连续 3 轮没有新信息：停止
3. 连续 3 轮 inspect：停止或强制切换 ToT
4. 同一个 tool + args 调用超过 2 次：拒绝执行
5. edit 前 confidence 必须足够
6. test 失败后不能重复运行同一个测试，除非中间发生过 edit
7. 没有读相关文件前，不能直接 edit
8. 遇到危险命令，停止并请求用户确认
```

例如：

```ts
function shouldStopOrChangeStrategy(state: AgentState, current: HistoryItem) {
  if (state.noProgressRounds >= 3) {
    return {
      type: "stop",
      reason: "连续多轮没有新进展"
    };
  }

  if (isStuckInspecting(state.history)) {
    return {
      type: "change_strategy",
      nextMode: "tot",
      reason: "连续 inspect，切换到 ToT 生成新假设"
    };
  }

  if (isRepeatedAction(state.history, current)) {
    return {
      type: "reject_action",
      reason: "重复工具调用，需要换一个动作"
    };
  }

  return {
    type: "continue"
  };
}
```

## 12.12 最小可用结构

最终可以整理成这个结构：

```txt
User Task
  ↓
Build Context
  ↓
Planner 输出 intent / tool / args / confidence
  ↓
Loop Guard 检查是否重复 / 是否空转 / 是否危险
  ↓
Executor 调用工具
  ↓
Observer 整理工具结果
  ↓
Update State
  ↓
Reviewer 判断是否完成
  ↓
继续 / 换策略 / 停止 / 最终回答
```

也就是：

```txt
模型负责判断
系统负责记录状态
Loop Guard 负责防止模型空转
工具负责提供真实反馈
Reviewer 负责判断是否完成
```

## 12.13 推荐文件结构

```txt
agent/
├── loop.ts          // 主循环
├── planner.ts       // 输出 intent / tool / args / confidence
├── tool-runner.ts   // executor + observer
├── reviewer.ts      // 判断是否完成
├── guard.ts         // 防重复、防空转、防危险操作
├── state.ts         // 保存 history / round / progress
├── context.ts       // 从 state 构造给模型的 context
└── modes.ts         // react / tot / aot / direct 切换逻辑
```

每个文件职责：

```txt
loop.ts
控制 while 循环

planner.ts
让模型决定下一步 intent

tool-runner.ts
执行工具并整理 observation

reviewer.ts
判断任务有没有完成

guard.ts
防止重复调用、连续 inspect、危险命令

state.ts
记录完整历史

context.ts
压缩 state，构造给模型看的 context

modes.ts
根据状态切换 ReAct / ToT / AoT
```

## 12.14 一句话总结

```txt
更智能的 Agent，不是让模型一直“想”，而是让系统统计每一轮有没有真实进展。
```

关键点是：

```txt
intent 记录这一轮想做什么
state 保存完整历史
context 给模型看压缩后的状态
guard 检测重复和空转
confidence 判断当前结论可信不可信
mode switch 在卡住时切换 ReAct / ToT / AoT
```

最终目标：

```txt
防止模型一直 inspect
防止模型重复查同一个文件
防止模型无证据改代码
防止模型测试失败后原地重试
防止 Agent 陷入无效循环
```

最短记法：

```txt
不要看模型嘴上说了什么
要看 state 有没有变化
```
