# AI Agent、CoT、ReAct、ToT、AoT 介绍与使用方式

## 1. 总览

AI Agent 不是某一种单独的推理方法，而是一个系统。

一个 Agent 通常由这些部分组成：

```txt
LLM 模型
+ 工具 Tools
+ 状态 State
+ 上下文 Context
+ 记忆 Memory
+ 规划 Planner
+ 执行循环 Loop
+ 反馈 / 验证机制
```

常见的推理 / 执行模式有：

```txt
CoT   = Chain of Thought，思维链
ReAct = Reason + Act，边想边行动
ToT   = Tree of Thoughts，思维树
AoT   = Algorithm of Thoughts，算法式思维
```

它们不是互斥关系。

更准确地说：

```txt
CoT   是基础推理方式
ReAct 是工具调用循环
ToT   是多路径搜索
AoT   是流程化 / 算法化约束
Agent 是把这些组合起来完成任务的系统
```

---

## 2. CoT：Chain of Thought

CoT 是“按步骤推理”。

普通回答是：

```txt
答案是 A。
```

CoT 风格是：

```txt
1. 先分析问题
2. 再拆分条件
3. 再一步步推导
4. 最后给出结论
```

CoT 适合：

```txt
逻辑推理
数学题
代码分析
Bug 原因判断
复杂需求拆解
```

在 Agent 里，CoT 通常是最底层能力。

比如：

```txt
ReAct 的每一次 Reason 可以用 CoT
ToT 的每一条分支可以用 CoT
AoT 的每一个步骤也可以用 CoT
```

但是在真正写 Code Agent 时，CoT 不一定需要单独做成一个模块。

因为 ReAct 里面本身就有 Reason。

```txt
Reason -> Act -> Observe
```

这里的 Reason 本身就可以包含 CoT。

所以工程上更推荐：

```txt
CoT 不单独暴露
CoT 作为 ReAct / ToT / AoT 内部的推理能力
```

---

## 3. ReAct：Reason + Act

ReAct 是：

```txt
Reason -> Act -> Observe -> Reason -> Act -> Observe
```

中文理解：

```txt
先想一下
然后调用工具
看工具结果
再继续想
再继续调用工具
直到完成任务
```

例如用户说：

```txt
帮我修复登录 bug。
```

ReAct 流程可能是：

```txt
Reason：我需要先找到登录相关代码
Act：search_files("login")
Observe：找到 login.ts 和 auth.ts

Reason：需要读取 auth.ts
Act：read_file("auth.ts")
Observe：发现 token 判断逻辑有问题

Reason：修改 token 判断逻辑
Act：edit_file("auth.ts")
Observe：修改成功

Reason：需要运行测试
Act：run_test()
Observe：测试通过

Final：输出修复总结
```

ReAct 适合：

```txt
读文件
搜代码
改代码
运行命令
查资料
操作浏览器
调用接口
多轮工具任务
```

Code Agent 最核心的模式通常就是 ReAct。

因为写代码不是一次性回答，而是：

```txt
搜索 -> 阅读 -> 修改 -> 验证 -> 再修改
```

这天然就是 ReAct。

---

## 4. Observe 是什么

Observe 指的是：

```txt
工具执行之后返回给模型的结果
```

也就是：

```txt
Act 调用了工具
Observe 接收工具结果
```

例如：

```txt
Act：
read_file("login.ts")

Observe：
文件内容是：
function login() {
  ...
}
```

这里的文件内容就是 Observe。

再比如：

```txt
Act：
run_cmd("npm test")

Observe：
测试失败：
Expected true but received false
auth.test.ts:24
```

这里的测试失败信息也是 Observe。

Observe 不是模型自己想出来的。

它是工具 / 环境返回的真实结果。

常见 Observe：

```txt
read_file 返回的文件内容
search_files 返回的搜索结果
run_cmd 返回的 stdout / stderr
edit_file 返回的修改结果
browser 返回的页面内容
API 返回的数据
```

一句话：

```txt
Observe = Agent 调用工具之后看到的真实反馈
```

---

## 5. ToT：Tree of Thoughts

ToT 是“多条思路一起探索”。

CoT 是一条链：

```txt
A -> B -> C -> 答案
```

ToT 是一棵树：

```txt
          问题
       /   |   \
    方案A 方案B 方案C
      |     |     |
    继续   继续   继续
```

例如用户说：

```txt
帮我设计一个 Code Agent 架构。
```

ToT 不会直接给一个答案，而是先生成多个方案：

```txt
方案 A：最简单 ReAct Agent
方案 B：Plan-and-Execute Agent
方案 C：多 Agent 编排系统
```

然后比较：

```txt
实现成本
稳定性
扩展性
适合当前项目程度
```

最后选择一个最合适的方案。

ToT 适合：

```txt
架构设计
多方案选择
复杂规划
算法设计
重构方案
产品方案
技术选型
多个可能原因的 Bug
```

ToT 和 ReAct 可以组合。

```txt
ToT 负责选路线
ReAct 负责走路线
```

例如：

```txt
ToT 生成 3 个方案
每个方案内部用 ReAct 去查资料 / 看代码 / 验证
最后比较哪个方案更好
```

结构类似：

```txt
ToT
├── 方案 A
│   └── ReAct 执行验证
├── 方案 B
│   └── ReAct 执行验证
└── 方案 C
    └── ReAct 执行验证
```

---

## 6. AoT：Algorithm of Thoughts

AoT 是“按算法流程思考”。

它不是随便一步步想，而是有固定流程。

例如修 bug：

```txt
1. 收集错误信息
2. 搜索相关代码
3. 建立假设
4. 验证假设
5. 选择最可能原因
6. 最小修改
7. 运行测试
8. 如果失败，回到第 3 步
```

这就是算法化流程。

AoT 的核心是：

```txt
不是自由发挥
而是按一个稳定流程执行
```

AoT 适合：

```txt
固定工作流
代码修复流程
安全检查流程
测试修复流程
批量任务流程
长期可复用任务
```

在工程 Agent 里，AoT 有时候也会被理解成：

```txt
Ahead-of-Time Planning
```

也就是：

```txt
先把任务编译成计划 / 图结构
再执行
```

例如用户意图：

```ts
[
  { intent: "read_file", target: "login.ts" },
  { intent: "read_file", target: "auth.ts" },
  { intent: "analyze", scope: "all" }
]
```

可以转换成执行图：

```ts
{
  nodes: [
    {
      id: "A",
      tool: "read_file",
      args: { path: "login.ts" },
      deps: []
    },
    {
      id: "B",
      tool: "read_file",
      args: { path: "auth.ts" },
      deps: []
    },
    {
      id: "C",
      tool: "analyze",
      args: { scope: "all" },
      deps: ["A", "B"]
    }
  ]
}
```

这个更像：

```txt
任务图
执行图
DAG Planning
Action Graph
```

所以要注意：

```txt
Algorithm of Thoughts = 算法式思维
Ahead-of-Time Planning = 提前生成执行计划 / 执行图
```

工程里有时候都会被简称成 AoT，但含义不完全一样。

## 7. 它们是平级、互相转换，还是互相嵌套？

它们不是严格平级，也不是简单互相转换。

更准确的是：

```txt
概念上可以平级比较
实现上可以互相嵌套
工程里可以互相组合
某些场景下可以互相转化
```

从概念分类上，可以这样比较：

| 模式    | 核心作用      |
| ----- | --------- |
| CoT   | 一步步推理     |
| ReAct | 推理 + 工具调用 |
| ToT   | 多路径思考     |
| AoT   | 算法化流程     |

但是在真正的 Agent 里，它们通常是嵌套的。

例如：

```txt
AoT 外层控制流程
  -> 每一步用 ReAct 调工具
      -> ReAct 的 Reason 里用 CoT 推理
```

或者：

```txt
ToT 生成多个方案
  -> 每个方案内部用 ReAct 验证
      -> 每次 Reason 用 CoT
```

所以不要理解成：

```txt
CoT、ReAct、ToT、AoT 四选一
```

应该理解成：

```txt
CoT 是基础思考能力
ReAct 是行动循环
ToT 是多方案搜索
AoT 是流程约束
Agent 是组合系统
```

## 8. Code Agent 推荐组合

### 8.1 简单问题

推荐：

```txt
Direct / CoT
```

适合：

```txt
解释一段代码
分析一个函数
判断一个小 bug
```

---

### 8.2 需要操作代码

推荐：

```txt
ReAct
```

适合：

```txt
搜索文件
读取文件
修改文件
运行测试
```

---

### 8.3 需要多个方案

推荐：

```txt
ToT + ReAct
```

适合：

```txt
设计架构
做技术选型
重构大型模块
多个可能原因的 bug
```

---

### 8.4 固定工作流

推荐：

```txt
AoT + ReAct
```

适合：

```txt
修 bug 流程
代码审查流程
测试修复流程
发布检查流程
```

---

### 8.5 最实用的组合

```txt
默认使用 ReAct
复杂方案使用 ToT
固定流程使用 AoT
内部推理使用 CoT
```

也就是：

```txt
普通任务：ReAct
复杂决策：ToT + ReAct
稳定流程：AoT + ReAct
基础推理：CoT
```

## 9. 最小架构示意

```txt
用户输入
  |
  v
Router 判断任务类型
  |
  |-- 简单解释 -> Direct / CoT
  |
  |-- 代码操作 -> ReAct
  |
  |-- 多方案设计 -> ToT
  |
  |-- 固定流程 -> AoT
  |
  v
执行 / 调工具 / 验证
  |
  v
最终回答
```

也可以设计成：

```txt
User Input
  ↓
Build Context
  ↓
Router
  ↓
Planner
  ↓
Tool Runner
  ↓
Observer
  ↓
Reviewer
  ↓
Final / Continue
```

## 10. 更智能的思考：Intent、Context 与防止无效推理

真正写 Agent 时，还要考虑一个问题：

```txt
模型会不会一直原地打转？
```

比如模型一直说：

```txt
我再检查一下
我继续分析一下
我再看看问题在哪里
```

但实际上：

```txt
没有读新文件
没有查新信息
没有修改代码
没有运行测试
没有产生新结论
```

这就是无效推理。

所以 Agent 需要记录每一轮的动作状态。

核心原则是：

```txt
不要只看模型说了什么
要看这一轮有没有产生新的状态变化
```

可以记录这些东西：

```ts
type TurnState = {
  readFiles: string[];
  searchedKeywords: string[];
  editedFiles: string[];
  executedCommands: string[];
  newFindings: string[];
};
```

每一轮结束后判断：

```txt
这一轮有没有新搜索？
这一轮有没有新读取？
这一轮有没有新修改？
这一轮有没有新验证？
这一轮有没有新结论？
```

如果都没有，就说明 Agent 可能在空转。

这时候可以强制它：

```txt
要么调用工具
要么输出结论
要么说明无法继续
不要继续空想
```

## 11. Intent 是什么

Intent 是“用户想让 Agent 做什么”。

例如用户说：

```txt
帮我看看这个 bug。
```

Router 可以判断成：

```ts
{
  intent: "debug",
  needsTool: true,
  mode: "react"
}
```

用户说：

```txt
解释一下这段代码。
```

可以判断成：

```ts
{
  intent: "explain",
  needsTool: false,
  mode: "direct"
}
```

用户说：

```txt
帮我设计一个 Code Agent 架构。
```

可以判断成：

```ts
{
  intent: "architecture_design",
  needsTool: false,
  mode: "tot"
}
```

用户说：

```txt
按固定流程帮我修复测试。
```

可以判断成：

```ts
{
  intent: "fix_test",
  needsTool: true,
  mode: "aot_react"
}
```

Intent 的作用是：

```txt
决定用什么模式
决定需不需要工具
决定要不要规划
决定是否进入循环
决定何时停止
```

## 12. Context 是什么

Context 是 Agent 当前能看到的信息。

包括：

```txt
用户输入
历史对话
系统提示词
项目路径
已读文件
工具结果
当前任务状态
记忆 Memory
临时计划
错误信息
测试结果
```

Context 很重要，因为模型不是凭空执行任务。

它每一步判断都依赖当前 Context。

例如 Code Agent 修 bug 时，Context 可能是：

```txt
用户说登录失败
搜索到了 login.ts
读取了 auth.ts
发现 token 判断逻辑
测试报错在 auth.test.ts:24
已经修改过 auth.ts 一次
npm test 仍然失败
```

这些信息会影响下一步动作。

所以 Agent 不能只保存 messages。

最好还要保存结构化状态。

例如：

```ts
type AgentState = {
  goal: string;
  intent: string;
  cwd: string;
  readFiles: string[];
  editedFiles: string[];
  commands: string[];
  findings: string[];
  errors: string[];
  plan: PlanStep[];
};
```

## 13. 最小 Code Agent 模式选择

可以简单设计成：

```ts
type AgentMode =
  | "direct"
  | "react"
  | "tot"
  | "aot_react";
```

Router 示例：

```ts
function route(input: string): AgentMode {
  if (input.includes("解释") || input.includes("是什么")) {
    return "direct";
  }

  if (input.includes("方案") || input.includes("架构") || input.includes("设计")) {
    return "tot";
  }

  if (input.includes("按流程") || input.includes("固定流程")) {
    return "aot_react";
  }

  if (input.includes("修复") || input.includes("bug") || input.includes("报错")) {
    return "react";
  }

  return "direct";
}
```

这不是最终生产级方案，但足够理解核心思想。

## 14. 总结

一句话理解：

```txt
CoT   = 怎么想
ReAct = 想完怎么行动
ToT   = 同时想多个方案
AoT   = 按固定流程想和做
Agent = 把模型、工具、状态、循环、验证组合起来
```

更工程化一点：

```txt
CoT   是推理能力
ReAct 是工具循环
ToT   是搜索策略
AoT   是流程约束
Agent 是执行系统
```

Code Agent 最推荐的默认结构是：

```txt
Router
  ↓
选择模式
  ↓
ReAct / ToT / AoT
  ↓
Tool Runner
  ↓
Observe
  ↓
State Update
  ↓
Loop or Final
```

最实用结论：

```txt
普通任务用 ReAct
复杂决策加 ToT
固定流程加 AoT
内部推理默认就是 CoT
```
