# 混合检索：ES + Milvus + Rerank + LLM

## 1. 一句话理解

混合检索可以理解为：

> ES 负责“这个词有没有出现”，Milvus 负责“这个意思像不像”，Rerank 负责“到底哪个最相关”。

实际工程里通常不是二选一，而是：

```text
ES 关键词召回
+ Milvus 向量召回
+ Rerank 重排
+ LLM 生成回答
```

这样可以同时解决两个问题：

```text
只用 ES：用户换个说法，可能搜不到
只用 Milvus：语义相似，但可能不够精确
```

---

## 2. ES 和 Milvus 各自适合什么

### ES：关键词检索

ES 更适合找“字面上出现过的词”。

适合场景：

```text
函数名
文件名
错误码
API 名
类名
配置项
专业术语
人名
项目名
```

例如：

```text
AgentSession
OPENAI_BASE_URL
401 unauthorized
v-hasPermission
@elastic/elasticsearch
```

ES 的优势是精确。

只要用户问题里包含这些关键词，ES 很容易命中相关内容。

---

### Milvus：语义检索

Milvus 更适合找“意思相近”的内容。

适合场景：

```text
用户换了一种说法
用户描述不精确
用户没有说出原文关键词
用户只描述现象
概念相近但字面不同
```

例如用户问：

```text
为什么模型接口一直说我没权限？
```

虽然问题里没有：

```text
401
unauthorized
OPENAI_BASE_URL
agentrouter
```

但 Milvus 仍然可能找到：

```text
API key 无效
认证失败
unauthorized client detected
bearer token
baseURL 配置错误
```

Milvus 的优势是语义泛化。

---

## 3. 为什么要结合使用

### 只用 ES 的问题

ES 主要看关键词。

比如数据库里有一条笔记：

```text
今天夜跑 6 公里，状态不错，拉伸后身体缓过来了。
```

用户问：

```text
我最近运动恢复怎么样？
```

如果原文没有“恢复”这个词，而是写“身体缓过来了”，ES 可能就不一定搜得准。

因为 ES 主要匹配：

```text
最近
运动
恢复
怎么样
```

如果这些词没有直接出现，命中效果就会下降。

---

### 只用 Milvus 的问题

Milvus 会根据语义相似度找内容。

用户问：

```text
我最近运动恢复怎么样？
```

Milvus 可能找到：

```text
今天夜跑 6 公里，状态不错，拉伸后恢复很快。
最近膝盖有点不舒服，运动后需要冰敷。
周六沿江骑行 20 公里，带上水和修车工具。
```

前两条很相关。

但第三条只是“运动相关”，不一定真的能回答“恢复情况”。

所以 Milvus 容易召回一些语义上接近，但实际问题相关性不够强的内容。

---

## 4. 混合检索的整体流程

完整流程一般是：

```text
用户问题
  ↓
ES 关键词召回
  ↓
Milvus 向量召回
  ↓
合并结果
  ↓
去重
  ↓
Rerank 重排
  ↓
取 Top K
  ↓
交给 LLM 生成回答
```

也就是：

```text
ES 负责精确词
Milvus 负责模糊语义
Rerank 负责最终排序
LLM 负责根据资料组织答案
```

## 5. 笔记场景示例

假设数据库里有几条笔记：

```text
1. 今天夜跑 6 公里，状态不错，拉伸后恢复很快。
2. 周六沿江骑行 20 公里，带上水和修车工具。
3. 下雨天在家看书，整理本周笔记并做晚餐。
4. 最近膝盖有点不舒服，运动后需要冰敷。
```

用户问：

```text
我最近运动恢复怎么样？
```

### ES 可能找到

ES 会根据关键词匹配：

```text
最近
运动
恢复
怎么样
```

它可能找到：

```text
4. 最近膝盖有点不舒服，运动后需要冰敷。
1. 今天夜跑 6 公里，状态不错，拉伸后恢复很快。
```

因为这些内容里有：

```text
最近
运动
恢复
```

---

### Milvus 可能找到

Milvus 会根据语义理解：

```text
运动表现
身体状态
恢复情况
疲劳
拉伸
伤痛
```

它可能找到：

```text
1. 今天夜跑 6 公里，状态不错，拉伸后恢复很快。
4. 最近膝盖有点不舒服，运动后需要冰敷。
2. 周六沿江骑行 20 公里，带上水和修车工具。
```

---

### Rerank 后可能留下

最终经过重排，可能得到：

```text
Top 1：今天夜跑 6 公里，状态不错，拉伸后恢复很快。
Top 2：最近膝盖有点不舒服，运动后需要冰敷。
Top 3：周六沿江骑行 20 公里，带上水和修车工具。
```

然后交给 LLM 回答：

```text
你最近整体运动状态不错，夜跑距离提升到 6 公里，并且拉伸后恢复较快。

不过也有膝盖不适的记录，说明运动后需要注意恢复和保护。短期内可以适当降低强度，继续观察膝盖状态。
```

## 6. 代码文档场景示例

假设知识库里有这些内容：

```text
1. OPENAI_BASE_URL 配置错误会导致 401 unauthorized。
2. API key 无效时会返回 unauthorized client detected。
3. Claude Sonnet 模型需要兼容 OpenAI 格式的代理服务。
4. 认证失败通常和 bearer token、baseURL、模型名有关。
```

用户问：

```text
为什么我 agentrouter 调 Claude 报 401？
```

### ES 很有用

因为用户问题里有明确关键词：

```text
agentrouter
Claude
401
```

ES 可以精确命中：

```text
401 unauthorized
agentrouter
OPENAI_BASE_URL
```

这种场景 ES 很强。

---

### Milvus 也有用

如果用户换个问法：

```text
为什么模型接口一直说我没权限？
```

这句话里没有：

```text
401
unauthorized
OPENAI_BASE_URL
agentrouter
```

ES 可能搜得不够准。

但 Milvus 可以根据语义找到：

```text
API key 无效
认证失败
unauthorized client detected
bearer token
baseURL 配置错误
```

所以两者结合更稳定。

## 7. 伪代码

```ts
async function hybridRetrieve(question: string) {
  const esResults = await searchFromES(question);
  const vectorResults = await searchFromMilvus(question);

  const merged = mergeAndDeduplicate([
    ...esResults.map(item => ({ ...item, source: 'es' })),
    ...vectorResults.map(item => ({ ...item, source: 'milvus' }))
  ]);

  const reranked = await rerank(question, merged);

  return reranked.slice(0, 5);
}
```

然后把结果交给大模型：

```ts
const docs = await hybridRetrieve('我最近运动恢复怎么样？');

const context = docs
  .map((doc, i) => `资料 ${i + 1}\n${doc.text}`)
  .join('\n\n');

const answer = await callLLM(`
你只能根据资料回答。

资料：
${context}

问题：
我最近运动恢复怎么样？
`);
```

## 8. 工程里的实际结构

可以理解成四层：

```text
1. 召回层
   - ES 查关键词
   - Milvus 查语义

2. 合并层
   - 合并两边结果
   - 去重
   - 标记来源

3. 排序层
   - Rerank 根据问题重新打分
   - 选出最相关的 Top K

4. 生成层
   - 把 Top K 文档作为 context
   - 让 LLM 根据资料回答
```

---

## 9. 为什么需要 Rerank

ES 和 Milvus 都只是“召回”。

召回的目标是：

```text
宁可多找一点，也不要漏掉重要内容
```

但是召回结果里会有噪声。

Rerank 的作用是：

```text
把真正能回答当前问题的内容排到前面
把只是表面相关的内容排到后面
```

例如用户问：

```text
我最近运动恢复怎么样？
```

Milvus 可能召回：

```text
周六沿江骑行 20 公里，带上水和修车工具。
```

这条和“运动”相关，但和“恢复情况”关系不强。

Rerank 会把它排到后面。

---

## 10. 总结

混合检索的价值是：

```text
ES 兜住精确关键词
Milvus 兜住语义相似
Rerank 负责最终相关性判断
LLM 负责组织自然语言答案
```

最终目标是：

```text
用户说得精确时，能搜到。
用户说得模糊时，也能搜到。
搜到之后，还能把最相关的内容排在前面。
```

一句话总结：

> ES 管“词”，Milvus 管“意思”，Rerank 管“相关性”，LLM 管“回答”。
